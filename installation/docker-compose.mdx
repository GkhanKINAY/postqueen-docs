## Docker Compose

This guide assumes that you have docker installed, with a reasonable amount of resources to run PostQueen. This Docker Compose setup has been tested with;

- Virtual Machine, Ubuntu 24.04, 2Gb RAM, 2 vCPUs.

<Snippet file="installation-pre-reqs.mdx" />

### Configuration uses environment variables

The docker containers for PostQueen are entirely configured with environment variables.

- **Option A** - environment variables in your `docker-compose.yml` file
- **Option B** - environment variables in a `postqueen.env` file mounted in `/config` for the PostQueen container only
- **Option C** - environment variables in a `.env` file next to your `docker-compose.yml` file (not recommended).

... or a mixture of the above options!

## The `docker-compose.yaml` file

The full, up-to-date Docker Compose file is maintained in the
[GkhanKINAY/postqueen-docker-compose](https://github.com/GkhanKINAY/postqueen-docker-compose)
repository. Cloning that repository (see the steps above) gives you the
`docker-compose.yaml` together with the `dynamicconfig` directory that the
Temporal service mounts, so there is nothing to copy by hand.

<Note>
Always pull the file from the repository rather than copying a snapshot — the
services, images, and environment variables change between releases, and the
repository is the canonical source.
</Note>

## How to use docker compose

From the cloned repository directory, run `docker compose up` to start the services.

<Warning>
**Note** When you change variables, you must run `docker compose down` and
then `docker compose up` to recreate these containers with these updated
variables.
</Warning>

Look through the logs for startup errors, and if you have problems, check out the [support](/support) page.

If everything looks good, then you can access the PostQueen web interface at https://postqueen.your-server.com

## Backups

Not every volume in the stack needs a backup. These hold real data:

- `postgres-volume` — the main PostQueen database (users, channels, posts).
- `temporal-postgres-data` — Temporal's database, including in-flight scheduled posts.
- `postqueen-uploads` — uploaded media, when `STORAGE_PROVIDER` is `local`.
- `postqueen-config` — your `postqueen.env` file, if you use Option B.

`postqueen-redis-data` and `temporal-elasticsearch-data` are a cache and a
search index — both are rebuilt automatically and do not need backups.

To back up a volume, archive it with a throwaway container. Compose prefixes
volume names with the project (directory) name, so check `docker volume ls`
for the exact name:

```
docker run --rm -v postqueen-docker-compose_postgres-volume:/data -v "$(pwd)":/backup alpine tar czf /backup/postgres-volume.tar.gz -C /data .
```

Repeat for each volume listed above.

<Warning>
`docker compose down -v` deletes all volumes — your database, scheduled
posts, and uploads included. Use plain `docker compose down` unless you
really mean to wipe the installation.
</Warning>
